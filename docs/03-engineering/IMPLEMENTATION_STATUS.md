| **Engineering Notes** | System Health slice uses the existing Actuator `/actuator/health` (ADR-015); no first-party health API endpoint was added. Authentication follows ADR-018 (refresh token in httpOnly cookie, access token in body); logout requires the access token per API_SPEC §5.4, refresh does not per §5.3. Login rate limiting (FR-AUTH-008, `429`) is a SHOULD and is deferred to a hardening pass. The authorization foundation is reusable infrastructure only — ownership is enforced in the service layer via a shared guard (SECURITY §5, BR-004), non-owned resources answer `404` (never `403`), and no role/permission model was introduced (RBAC remains future per SECURITY §19). No method-security annotations were added: no frozen document specifies them, and ownership is not expressible as a role check. User Profile: the `user` aggregate stays in `auth` and the `users` module reaches it through a published, DTO-only `UserAccountService` (ARCHITECTURE §5.1 forbids cross-module internals access). No migration was needed — V2 already created `professional_details` and `preferences`. `UserResponse` gained the `professionalDetails`/`preferences` fields API_SPEC §17.1 already specified (additive, non-breaking per §20). VR-003 lengths are externalized as `users.profile.*` because the SRS marks them [Assumption] and they are not finalized; `preferences` is stored as an opaque jsonb blob since no document defines its keys. Document Management: implemented per API_SPEC §8 with the lifecycle stopping at `UPLOADED` (OCR/processing transitions out of scope). No `document_content` table yet — it is populated only by OCR/extraction and its full-text index powers Search, both later slices. No document **edit** endpoint exists in API_SPEC §8, so none was built. VR-005 allowed types/max size are `documents.*` config ([Assumption], not finalized). The optional 409-on-archived-client upload policy (API_SPEC §8.1) was not implemented. Storage bytes reach Supabase through the ADR-008 port; provider REST calls are confined to `SupabaseStorageAdapter` (no provider type escapes it); ITs use an in-memory port. OCR Processing: extraction runs **synchronously-with-status** (ADR-013 — no worker/queue/scheduler); upload triggers it inline using the in-memory bytes and returns the initial `UPLOADED` snapshot (API_SPEC §8.1), with the terminal state observed via `ocr-status` (§9.1, §2.11 poll). Native-first: PDFBox extracts embedded PDF text; insufficient text or images fall back to the OCR port (Google Vision, adapter confined, in-memory port in tests). `document_content` is written **only** by successful extraction (native or OCR). **Gap:** no documented endpoint exposes the extracted text, so no extracted-text viewer was built — the frontend shows OCR status/method/quality/failure only; exposing the text would need a new API_SPEC endpoint (stop-and-ADR). The Failed→Processing retry transition exists in SRS §7.1 but has no documented trigger endpoint, so no retry was implemented. AI Summary: implemented per API_SPEC §10 (generate/get/edit) on the resolved provider ([ADR-003](../01-architecture/decisions/ADR-003-AI-Provider-Abstraction.md) → Anthropic). Generation runs **synchronously-with-status** (ADR-013) and the provider call sits **outside** the DB transaction (ADR-010); the `AIRequest` lifecycle (`REQUESTED→IN_PROGRESS→COMPLETED |FAILED`) is observable via the`GET` poll (§2.11). The grounded prompt is composed centrally and channel-separated (system instructions vs. document data — the injection defense, AI_ARCHITECTURE §8/SECURITY §10); document text is treated as data. The model is reached **only** through the domain-owned `AiPort`; the Anthropic adapter (Spring`RestClient` against the Messages API, mirroring the Vision/Supabase adapters) is the sole place aware of the provider — no provider SDK, request, or response type escapes it, and it is absent from the test profile (an in-memory port stands in, so tests never contact a real provider). The composed prompt is not persisted (AI_ARCHITECTURE §14); `AIRequest.prompt` is null for summaries. An empty model response is an output-validation failure (FAILED, recorded), never a fabricated success (§11). **Judgment call:** the synchronous `POST` returns `201` for the existing summary when `regenerate` is absent (idempotent) — API_SPEC §10.1 lists only `201`/`202` for POST, so `201` is used for both fresh and pre-existing results rather than inventing a `200` path. `SUMMARY_GENERATED` activity emission (API_SPEC §10.1 note) is now wired via the Activity Timeline slice (below). Only `SUMMARY` is implemented; the `AIRequest.type`/`status` CHECKs carry the full documented value sets so Chat/Email/Report need no schema change. Activity Timeline: implemented per API_SPEC §15 / DATABASE §5.8 — a read-only `GET /api/v1/activities` over an append-only `activity` table (no update/delete path, no `updated_at`; BR-016/FR-TMLN-004). Reads are owner-scoped by`user_id` (BR-006) at the query (no ownership logic duplicated); the optional `clientId` gives the per-client view, and a non-owned `clientId` simply yields an empty page (user-scoping already isolates, so no cross-user leak). The documented events for the built modules — `CLIENT_CREATED` (client create), `DOCUMENT_UPLOADED`/`DOCUMENT_DELETED` (document upload/delete), `SUMMARY_GENERATED` (summary generate) — are emitted through **one shared `ActivityService`**, so persistence lives in a single place and no module writes activity rows itself. Emissions inside a mutation's transaction (client create, document delete) commit atomically with it (DATABASE §11); those in the storage-first upload flow and the provider-outside-transaction summary flow commit immediately after (consistent with those flows' existing non-atomic design).`EMAIL_GENERATED`,`REPORT_CREATED`, and chat activity are **not** emitted — those modules are not built; nothing is invented beyond the documented set, and`action_type`has no DB CHECK (DATABASE §5.8) so a future capability needs no schema change. The summary activity is document-scoped (`client_id` null), so it appears in the account timeline but not the per-client view. No messaging/queue/scheduler/notification infrastructure was introduced — the timeline is a persistence + presentation feature only. **Observation (not changed here):** a wrong HTTP method on `/activities` (e.g. `POST`) currently surfaces as`500` via the shared catch-all handler rather than `405`; that is pre-existing, app-wide handler behavior, so a dedicated`405` RFC-7807 mapping is left as a general future improvement rather than an in-scope change. Global Search: implemented per API_SPEC §14 / DATABASE §9 — a read-only `GET /api/v1/search` doing PostgreSQL built-in full-text keyword search (`to_tsvector`/`websearch_to_tsquery`/`ts_rank`) over `document_content.extracted_text`, **reusing the existing `gin_document_content_extracted_text` index (V5)** — no new index, migration, or search infrastructure, and no semantic search or vector store. Owner-scoped at the query by the owning client's `user_id` (BR-005 — the same owner-scoping-at-the-query approach as the client list/timeline, no ownership logic duplicated); excludes `DELETED` documents (BR-013/FR-SRCH-004); only documents with extracted content (i.e. `READY`) can match via the `document_content` join (FR-SRCH-005). §14.1 defines no `sort` parameter, so results are relevance-ordered (`ts_rank`) with a deterministic `updated_at`/`id` tiebreak for stable pagination, and any incoming `sort` is stripped (a native full-text query cannot take dynamic sorting). **VR-006 semantics (corrected in this verification pass):** an **empty/blank query returns a helpful empty state — an empty page (200), not a 422** (VR-006, FR-SRCH-006, API_SPEC §14.1); only an **over-length** query is a `422` field error. The prior implementation wrongly returned `422` for a blank query; that was fixed to match the documented contract, with the service/controller/e2e tests updated accordingly. **Real-PostgreSQL/Hibernate hardening:** the epoch-millis projection column uses `cast(… as bigint)` rather than the PostgreSQL `::bigint` shorthand, because Hibernate can misread `::` as a named parameter. Snippet/`matchContext` are derived in Java from a bounded text excerpt (no highlighting markup); the query length bound is the only `[Assumption]` and is externalized as `search.max-query-length`. **Verification caveat:** the Docker-backed integration tests (`SearchPersistenceIT`, `SearchIT`) are **written but remain unexecuted in the local environment — no Docker/PostgreSQL runtime is available here** — so they are not claimed as passing; they run in CI. Unit/web tests pass. **Not implemented (documented refinement):** VR-006 also lists "disallowed control characters" as an invalid case; this is not enforced because `websearch_to_tsquery` accepts any input safely (never throws), so it is not a correctness risk — noted as a future refinement, not invented behavior. Report Generation: implemented per API_SPEC §13 / SRS §4.10 — `POST /documents/{id}/reports` generates, and `GET /reports` (owner-scoped, paged, optional `documentId`/`status` filters, newest-first) + `GET`/`PATCH`/`DELETE /reports/{id}` manage. Reports **orchestrate existing capabilities, they do not reimplement them**: the document's extracted text is fetched through the already-published `DocumentService.requireOwnedContentForAi` (the same accessor the AI-Summary slice uses — no ownership or content logic duplicated), and the report body is produced through the domain-owned `AiPort` with a centrally-composed, grounded, channel-separated **report** prompt (`ReportPromptBuilder`, mirroring `SummaryPromptBuilder`; document text is data, not instructions — AI_ARCHITECTURE §8/SECURITY §10). No provider SDK/type is added — the existing Anthropic adapter is reused, absent from the test profile (in-memory port). **Persistence-model decision (Decision Log 2026-07-23):** DATABASE §11 gives report generation its **own** transaction boundary ("insert/update `Report` + insert `Activity(REPORT_CREATED)`") distinct from the generic `AIRequest`/`AIOutput` AI flow, so a report persists as a dedicated `report` row with its content, **not** an `AIRequest`/`AIOutput` pair — no `AIRequest` is created for reports. SRS's conceptual "an AI request is created" is satisfied by the AI-port call; DATABASE is authoritative on persistence, so this was treated as a resolved interpretation, not a blocking gap. The provider call runs **outside** the DB transaction (ADR-010); on success the `Report` + `REPORT_CREATED` activity commit **atomically** in one transaction (`ReportLifecycleWriter`, DATABASE §11) — the only new `ActivityType`/event, emitted through the shared `ActivityService` (no module writes activity rows itself). `V8__reports.sql` adds the `report` table (FK `document_id`/`user_id`, `status` CHECK `DRAFT`/`SAVED`, owner + document indexes); reports are **hard-deleted** (DELETE removes the row) as API_SPEC §13.5 specifies no soft-delete for reports. Reused exceptions map to the documented statuses (document not `READY` → `409`; no extractable text → `422`; provider/empty-output → retryable `503`). A missing or over-length title → `422`; `PATCH` validates content (blank/over-length → `422`) and parses `status` (invalid → `422`); `report` title/content length bounds are externalized as `reports.*` config ([Assumption], not finalized). Export (FR-RPT-004) is a **client-side** download of the returned content (API_SPEC §13.4 states export is client-side — **no export endpoint**), so none was invented. The reports UI reuses the shared API client, pagination, layout, auth guard and React Query patterns; the document-embedded panel gates generation until the document is `READY` (BR-010) and does not fetch reports before then. **No** scheduled reports, notifications, email delivery, background jobs, dashboards, charts, exports beyond §13.4, or semantic/vector/RAG/chat capability was introduced. **Verification caveat:** the Docker-backed integration tests (`ReportPersistenceIT`, `ReportIT`) are **written but remain unexecuted locally — no Docker/PostgreSQL runtime is available here** — so they are **not** claimed as passing; they run in CI. Backend unit/web tests (248) and the frontend suite (108) pass. |

## 13. Update Rules

This document is **living** and MUST always reflect the current implementation state. Update it:

- **After every completed milestone** — mark the milestone Completed with its date; advance Current Milestone (§2, §6).
- **After every sprint** — roll the Current Sprint (§7) and add a Weekly Progress Log entry (§11).
- **After every ADR** — record it in the Decision Log (§10); update the Deferred Decisions Tracker (§5) if it resolves
  one.
- **After every major feature** — update the Module (§4) and Phase (§3) trackers and the Completion Checklist (§12).
- **After every production deployment** — update Project Summary (§2) and the relevant trackers.

Whenever any of the above happens, also refresh **Last Updated** and **Overall Progress**
in [Project Summary §2](#2-project-summary).

> This dashboard is expected to change constantly. If it ever disagrees with reality, **reality is right and the
> dashboard is stale** — fix it. It never overrides the frozen documents or
> the [IMPLEMENTATION_PLAN](./IMPLEMENTATION_PLAN.md);
> it only reports progress against them.
